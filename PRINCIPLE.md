# 指纹检测原理

本文逐项说明检测引擎校验的每个维度。核心思想一句话：**让字段之间互相约束，伪造单个字段必然破坏另一个字段的一致性。** 检测不依赖算法保密，而依赖这些可解释的物理/数学约束。

---

## 0. 字段分层模型

任何浏览器环境特征按"变化频率"分三层。检测的前提是搞清每个字段属于哪一层，因为不同层有不同的校验规则。

| 层 | 字段 | 同一会话内 | 跨会话/换设备 |
|----|------|-----------|--------------|
| 硬件层 | canvas hash+histogram、GPU、math 三角函数、屏幕分辨率、插件列表 | **必须恒定** | 才会变 |
| 环境层 | userAgent、时区、performance.timing、navigator 属性 | SPA 内恒定 | 可能变 |
| 行为层 | 点击/按键计数、鼠标轨迹、时间戳、停留时长 | **每次都变** | 必变 |

> 攻击方最常见的破绽：把硬件层字段当行为层随机生成。检测引擎据此识别。

---

## 1. 结构校验（零成本，零误杀）

最廉价的一道闸：字段格式不对直接判伪造。

- **histogram 必须恰好 256 个桶**：它是 canvas 像素 RGBA 通道值的频率分布，统计学上固定 256 个区间。多一个少一个都不是真实采集。
- **histogram 求和不变量**：所有桶求和 = 被统计的像素采样数，对固定尺寸 canvas（如 150×60）这个总和落在确定范围。求和异常 = 数据是拼凑的。
- **canvas hash 是 signed int32**：CRC32 结果，范围 `[-2^31, 2^31)`。超范围 = 伪造。
- **时间戳为合理毫秒级**：`start`/`end` 应接近当前服务器时间（容忍时钟漂移），偏差过大说明是离线生成的固定值。

---

## 2. 绑定一致性（TES 最关键的一招）

这是 TES 检测的灵魂，也是源项目作者踩坑最深的地方（README 明确记录："给 histogram 加 ±2 jitter 直接 BLOCKED"）。

**约束**：`canvas hash` 与 `histogramBins` 必须来自同一次真实 canvas 渲染。

- canvas hash = `CRC32(渲染输出)`，histogram = 同一渲染输出的像素分布。两者由同一段像素数据派生，存在**确定性绑定**。
- 真实浏览器在同一设备上，这对值**永远一致且可复现**。
- 攻击方若想伪装多设备，会随机 hash 但用同一套 histogram（或反之）——绑定立即被打破。

**检测实现**：服务端维护 `hash → histogram 指纹` 注册表。

```
首次见到某 hash → 记录其 histogram 摘要
再次见到同一 hash → histogram 摘要必须一致，否则判伪造
```

这条规则让"批量伪造设备"几乎不可能：要么所有号共享一对真实值（被关联分析抓），要么随机配对（被绑定校验抓）。

---

## 3. 会话稳定性

同一会话（同一用户、同一页面生命周期）内，硬件层与环境层字段不得突变。

- canvas hash、GPU、屏幕分辨率在一次会话内**不可能改变**——硬件不会中途换。
- `performance.timing` 在同一 SPA 页面内恒定：真实浏览器单页应用内导航不会重新触发 `navigationStart`。源项目专门处理了这点（"[191] 和 [197] 的 timing 完全相同"）。
- 检测实现：按会话 key 缓存首次上报的硬件字段，后续上报逐一比对，突变即加分。

---

## 4. 自动化痕迹

直接识别"这不是真人浏览器"。这些信号单独看可能误杀，组合看高度可信。

| 信号 | 含义 |
|------|------|
| `navigator.webdriver === true` | 显式自动化标志 |
| `window.phantom` / `__nightmare` / CDP 残留 | 无头框架注入痕迹 |
| GPU vendor 含 "SwiftShader"/"llvmpipe"/"Software" | 软件渲染 = 无真实 GPU = 服务器/容器 |
| 插件列表为空 + 桌面 UA | 真实桌面 Chrome 一般有 PDF 插件等 |
| UA 声称 Chrome 但缺 Chrome 特有 API | UA 与运行时能力矛盾 |
| 屏幕分辨率为 0 或非常规值 | 无真实显示器 |
| WebGL 扩展列表为空 | 真实 GPU 会暴露一组扩展 |

---

## 5. 行为合理性

把行为层数据和事件类型对照，检查是否符合人类操作物理规律。

- **PageLoad 事件应零交互**：页面刚加载，用户还没动，`clicks/keyPresses` 必须为 0。源项目验证过"[119] bootstrap interaction 全零"。非零 = 脚本伪造了交互。
- **提交事件应有人类节奏**：
  - 按键时间间隔 `keyPressTimeIntervals` 应有自然抖动（80~700ms），全相等或为 0 = 机器输入。
  - 交互数量关系：`intervals 数量 < keyPresses`（连续快打时部分间隔被合并），违反 = 伪造。
- **停留时长合理**：`timeToSubmit` 几十毫秒就提交完整表单 = 不是人类。

---

## 6. 时序连贯性

时间字段之间存在严格先后关系，违反即伪造。

- `start ≤ end`，且 `end - start`（停留/采集时长）非负且合理。
- `performance.timing` 内部顺序：`navigationStart ≤ fetchStart ≤ domainLookupStart ≤ connectStart ≤ responseStart ≤ domInteractive ≤ loadEventEnd`。源项目 `_gen_perf_timing` 正是按这个顺序构造的——检测方反过来校验这个顺序。
- `loadEventEnd` 应早于指纹 `start`（页面加载完才开始采集）。

---

## 7. 关联分析（需积累基线）

前面 6 项是单次请求内的自洽性校验，本项是跨请求的群体校验，用于抓批量行为。

**核心指标**：一个硬件指纹（canvas hash）关联了多少个不同账号/邮箱。

- 真实场景：一台设备 = 一个 canvas 指纹，绑定 1~3 个账号（家庭共用）属正常。
- 农场场景：同一 canvas hash 横跨几十上百个账号 → 批量注册信号。
- 源项目 README 自己承认了这个软肋："目前只有 1 组采集数据，所有号共享同一 canvas_hash，AWS 做关联分析可以识别出批量注册行为"——这正是检测方要利用的。

**检测实现**：维护 `hash → Set<accountId>` 表，超过阈值（如 5）开始加分，呈非线性增长。

> 关联分析需要时间窗口和基线，建议作为第二阶段上线，初期先靠 1-6 项（零误杀）。

## 风险分合成

每项命中累加权重分，最终归一到 0-100：

```
结构校验失败      → +40（基本可判定伪造）
绑定不一致        → +50（TES 核心，强信号）
会话突变          → +30
自动化痕迹（每项） → +15
行为不合理        → +20
时序矛盾          → +25
关联超阈值        → +10 起，按数量非线性增长
```

分级：`0-29 low / 30-69 mid / 70-100 high`。每个分数附带命中原因列表，便于人工复核与阈值调优。
